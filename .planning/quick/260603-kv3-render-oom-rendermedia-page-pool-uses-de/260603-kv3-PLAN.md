<objective>
The remotion-renderer step crashes at render start (frame 0) with:
`Render failed: Visited "http://localhost:3000/index.html" but got no response.`
Stack: `getPool` → `Promise.all` (index 2) → `makePage` → `gotoPageOrThrow`.

Confirmed root cause: `renderMedia` in render.ts sets NO `concurrency`, so Remotion
defaults to ~cores/2. On this host (16 cores → ~8 pages) with `enableMultiProcessOnLinux: true`,
each page is multiple Chrome processes decoding a video + a PNG overlay. With only
~7.5 GB RAM available, the page pool exhausts memory and Chrome tabs (index 2+)
crash → "got no response". `selectComposition` (1 page) succeeds; the multi-page
pool is what dies.

Fix: make render concurrency explicit and bounded via `REMOTION_CONCURRENCY`
(default a safe low value), and pass it from the orchestrator so the production
`/process` path uses it. This matches the pipeline's single-job, memory-bounded design.
</objective>

<diagnosis_evidence>
- /tmp/render-debug.log (manual re-run of the renderer on job 906c75b4): bundle OK, selectComposition OK (666 frames), Render 0% → crash in getPool/makePage.
- Host: `nproc`=16, `free -h` ≈ 1.1 GB free / 7.5 GB available.
- render.ts: renderMedia (≈L349-365) has NO `concurrency` option.
- orchestrator.ts: remotion-renderer step envVars (≈L100-130) — where to add REMOTION_CONCURRENCY.
</diagnosis_evidence>
