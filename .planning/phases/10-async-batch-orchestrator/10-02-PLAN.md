Per D-01, D-02, D-03: Create batch upload endpoint and batch status endpoint.

1. **Create services/api-server/src/routes/batch.ts:**

   a. **Imports:** Router from express, multer (reuse config pattern from process.ts), uuid v4, Zod schemas from batch.ts, queue from queue.ts, progress helpers from progress.ts, PIPELINE_DATA_DIR from constants.ts.

   b. **Multer configuration for batch upload:**
   - Use `multer.diskStorage` like process.ts — same tmp directory pattern
   - `upload.array("videos", MAX_BATCH_SIZE)` where MAX_BATCH_SIZE comes from env var or defaults to 10 (per D-10)
   - Same `fileFilter` as process.ts (only video/mp4)
   - Same `limits.fileSize` as process.ts (500MB per file)
   - Read MAX_BATCH_SIZE from `process.env.MAX_BATCH_SIZE` with default 10

   c. **POST /batch handler:**
   - Validate files exist (`req.files` must be array with length > 0)
   - Validate file count doesn't exceed MAX_BATCH_SIZE
   - Generate a `batchId` (UUID v4)
   - For each file in `req.files`:
     - Generate a `jobId` (UUID v4)
     - Create job directory: `PIPELINE_DATA_DIR/{jobId}/input/`
     - Move file from tmp to `PIPELINE_DATA_DIR/{jobId}/input/video.mp4`
     - Add BullMQ job to `videoQueue` with data: `{ jobId, batchId, filename: file.originalname, inputPath: /data/pipeline/{jobId}/input/video.mp4 }`
     - Call `addJobToBatch(batchId, jobId)` to store batch→job mapping in Redis
     - Call `updateJobProgress(jobId, { status: "queued", currentStep: "queued" })` for initial progress tracking
     - Collect { jobId, filename, status: "queued" } for each job
   - Return 200 with BatchResponseSchema-validated response: `{ batchId, jobs, createdAt: new Date().toISOString() }`
   - On error: return appropriate status codes (400, 413, 415, 500)

   d. **GET /batch/:batchId handler:**
   - Call `getBatchJobs(batchId)` to get array of jobIds
   - If no jobs found, return 404 `{ error: "Batch not found" }`
   - For each jobId, call `getJobProgress(jobId)` to get current status
   - Also check BullMQ job state via `videoQueue.getJob(jobId)` for authoritative status
   - Merge Redis progress data with BullMQ job state (BullMQ state takes precedence for status)
   - Build response: `{ batchId, jobs: [{ jobId, filename, status, currentStep, error }], completedCount, failedCount, totalCount }`
   - Return BatchStatusResponseSchema-validated response

   e. **Error handling:**
   - MulterError LIMIT_FILE_SIZE → 413
   - MulterError LIMIT_FILE_COUNT (exceeds MAX_BATCH_SIZE) → 413
   - UnsupportedMediaTypeError → 415
   - Generic errors → 500

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| External user → POST /batch | Untrusted input (multiple file uploads, large request bodies) |
| POST /batch → BullMQ queue | Internal trusted (job creation) |
| GET /batch/:batchId → Redis | Internal trusted (data retrieval) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-05 | Tampering | POST /batch file upload | mitigate | Multer fileFilter validates video/mp4 mimetype per existing D-01 pattern; MAX_BATCH_SIZE caps file count per D-10 |
| T-10-06 | Denial of Service | POST /batch | mitigate | MAX_BATCH_SIZE=10 prevents flooding; fileSize limit per file capped at 500MB |
| T-10-07 | Information Disclosure | GET /batch/:batchId | mitigate | batchId is UUID — unguessable; no authentication required for v1 per CONTEXT.md deferred items |
| T-10-08 | Spoofing | Batch ID enumeration | accept | UUID v4 batchIds are 128-bit random — enumeration impractical |
</threat_model>
