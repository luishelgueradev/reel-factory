1. **Create services/api-server/src/schemas/batch.ts:**
   - `BatchJobSchema` = z.object({ jobId: z.string().uuid(), filename: z.string(), status: z.enum(["queued", "active", "completed", "failed"]) })
   - `BatchResponseSchema` = z.object({ batchId: z.string().uuid(), jobs: z.array(BatchJobSchema), createdAt: z.string() })
   - `BatchStatusResponseSchema` = z.object({ batchId: z.string().uuid(), jobs: z.array(z.object({ jobId: z.string().uuid(), filename: z.string(), status: z.enum(["queued", "active", "completed", "failed"]), currentStep: z.string().optional(), error: z.string().optional() })), completedCount: z.number(), failedCount: z.number(), totalCount: z.number() })
   - Export all schema types: `BatchJob`, `BatchResponse`, `BatchStatusResponse`
   - Export `JOB_STATUS_VALUES` = `["queued", "active", "completed", "failed"] as const` for reuse

2. **Create services/api-server/src/progress.ts:**
   - Import Redis from `ioredis`
   - Import `createQueueConnection` from `./queue.js` (reuse the shared Redis connection)
   - Key pattern: `job:{jobId}` for progress hashes, `batch:{batchId}` for batch→jobId lists
   - Export `updateJobProgress(jobId: string, data: { currentStep?: string; status: string; error?: string })`: Sets fields on Redis hash `job:{jobId}` using `redis.hSet()` with fields: `currentStep`, `status`, `error`, `updatedAt` (ISO timestamp). Use `redis.expire()` to set 24-hour TTL per D-05 note about TTL to prevent stale data accumulation.
   - Export `getJobProgress(jobId: string)`: Returns `Promise<Record<string, string> | null>` using `redis.hGetAll(job:{jobId})`. Returns null if no hash exists.
   - Export `addJobToBatch(batchId: string, jobId: string)`: Adds jobId to Redis list `batch:{batchId}` using `redis.rPush()`. Set 25-hour TTL on the batch key (slightly longer than job TTL).
   - Export `getBatchJobs(batchId: string)`: Returns `Promise<string[]>` using `redis.lRange(batch:{batchId}, 0, -1)`.
   - For all operations, use the shared Redis connection from createQueueConnection (NOT creating new connections per call)

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| External user → POST /batch | Untrusted input (file uploads, request parameters) |
| API server → Redis | Internal trusted connection (Docker network) |
| BullMQ worker → Docker daemon | Internal trusted (sibling container orchestration) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-01 | Tampering | POST /batch | mitigate | Multer file validation (video/mp4 only) + MAX_BATCH_SIZE limit (D-10) prevents flood attacks |
| T-10-02 | Denial of Service | POST /batch | mitigate | MAX_BATCH_SIZE=10 limits per-request file count; BullMQ job queue prevents resource exhaustion |
| T-10-03 | Information Disclosure | Redis | accept | Redis bound to Docker internal network only; no external port exposure |
| T-10-04 | Tampering | Batch job data | mitigate | Zod schema validation on batch responses ensures type safety; Redis keys are namespaced |
</threat_model>
