<objective>
Fix the `@remotion/google-fonts` `loadFont` call that passes the options object in
the wrong positional argument, causing every font to throw and fall back to monospace
(generic fonts) in both the studio preview and production renders.
</objective>

<root_cause>
`@remotion/google-fonts` signature is `loadFont(style?, options?)` — first arg is the
font style (`"normal"`/`"italic"`), second is `{ weights?, subsets? }`. Both
`services/remotion-studio/src/fonts.ts:122` and
`services/remotion-renderer/src/fonts.ts:122` call:

    loader.loadFont({ subsets: ["latin", "latin-ext"] })

passing the options object into the `style` slot. The lib throws
`The font <X> does not have a style [object Object]`, the catch swallows it and returns
"monospace", so ALL fonts render generic. Confirmed via Chrome console on
`/preview` (SpaceGrotesk, Rubik, etc.).
</root_cause>

<tasks>
1. **Fix studio + renderer fonts.ts** — change both call sites to
   `loader.loadFont("normal", { subsets: ["latin", "latin-ext"] })`. Keep the two
   files in sync per CLAUDE.md renderer-sync convention.
2. **Rebuild & verify** — rebuild the studio editor + the `remotion-studio` and
   `remotion-renderer` docker images. Confirm the `does not have a style` throw is
   gone (console) and the selected font family applies in the preview.
</tasks>

<verification>
- No `[fonts] Failed to load font ... does not have a style [object Object]` warning in console.
- Selected font (e.g. SpaceGrotesk/Raleway) applies in the `/preview` player.
- Both fonts.ts files identical (sync convention).
</verification>
