<objective>
Turn the Studio Express server into the single public origin for rendering: replace the 501
`POST /api/render` stub with a real streaming proxy and add `GET /api/status/:jobId` and
`GET /api/result/:jobId`. This is the RENDER-01 submit path, the RENDER-02 poll relay, and the
RENDER-04 finished-video relay — all behind the Studio's existing basic auth, keeping api-server
internal on pipeline-net (D-03).

**Resolves the phase's one architectural seam (RESEARCH Open Question 2 / PATTERNS L79-89):**
`POST /process` (process.ts L96-165) is SYNCHRONOUS — it `await runPipeline(...)` and only returns
the jobId in the final body minutes later, which defeats live progress polling (RENDER-02). The
poll-friendly path is the **queue**: `POST /batch` (batch.ts L79-141) calls `videoQueue.add(...)`
(L112) and returns the jobId IMMEDIATELY with `status:"queued"`, seeding the `job:{id}` Redis
progress hash (batch.ts L123) that `GET /status/:jobId` reads. **Therefore `POST /api/render`
forwards the upload to `POST /batch` (field name `videos`), not the blocking `/process`** — this is
the only way RENDER-01 (submit) and RENDER-02 (live progress) both work end-to-end. The browser
gets the jobId from the `/batch` response (`jobs[0].jobId`) and starts polling.

Purpose: Studio becomes a self-contained video factory (RENDER-01); the jobId-up-front contract
makes live progress possible (RENDER-02); the finished MP4 is reachable proxied (RENDER-04).
Output: 3 proxy routes in server.ts + server.test.ts.
</objective>
