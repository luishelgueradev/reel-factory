<objective>
Make renderer font loading deterministic and offline-first so a transient `fonts.gstatic.com` blip
can neither abort nor silently degrade a 666-frame render (RENDER-05 â€” the highest-risk item). Today
`loadFont` (fonts.ts L109-131) ends on `monospace` on any failure and has NO timeout, so a stuck
gstatic fetch is bounded only by the 3h process timeout (hang path) and a degraded render looks broken.

Rewrite the fallback chain to: (1) local vendored woff2 via `@remotion/fonts` `loadFont({family,
url: staticFile(...)})` â€” offline, deterministic (D-10); (2) gstatic via the existing
`@remotion/google-fonts` loader as the network fallback; (3) a bundled default sans (Plus Jakarta
Sans / Inter) as the FINAL fallback â€” NEVER monospace (D-12). Wrap EVERY attempt in a ~10s
`withTimeout` race so a never-resolving load falls through instead of hanging (D-11).

**Three hazards this plan must respect (from MEMORY + AGENTS.md):**
1. **Version pin:** install `@remotion/fonts@4.0.457` (NOT latest 4.0.471) to match `remotion@4.0.457`
   â€” a mismatched @remotion/* version causes cryptic render failures (AGENTS.md Version Compatibility).
2. **Renderer-sync clobber hazard:** `fonts.ts` exists in BOTH studio and renderer; make the change in
   BOTH and re-run renderer vitest after any sync, or a later bulk `cp` silently reverts the fix.
3. **Image rebuild:** the renderer Dockerfile does not COPY public/ â€” vendored fonts only take effect
   after `COPY public/ public/` + an image rebuild (verified in Plan 05's smoke gate).

Purpose: the literal RENDER-05 guarantee â€” a render completes with a real sans even if gstatic is down.
Output: rewritten fonts.ts (both services), vendored woff2, Dockerfile COPY, fonts.test.ts.
</objective>
