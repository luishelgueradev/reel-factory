<objective>
Wire the three persistence fixes that make studio-saved config survive Docker rebuilds:

1. Create the git-tracked default template (D-06, D-07).
2. Fix the studio PUT /api/config write target to use ACTIVE_PIPELINE_CONFIG_PATH as the sole destination — eliminating the resolveConfigPath() non-determinism (D-04, D-05).
3. Add a startup seed-copy hook in api-server that hydrates the active config from the template on fresh install (D-06).
4. Align docker-compose.yml so api-server reads from the same persistent path the studio writes to, and remove the now-unneeded :ro studio volume (D-02).
5. COPY config/ into the api-server Docker image so the template is available at /app/config/ at runtime.

Purpose: Close the writer/reader split that caused studio-saved config to be "lost" on git clean or docker builds. After this plan the active config at ./pipeline/pipeline-config.json is the single source of truth — durable between builds because it lives on the host bind mount.

Output: config/pipeline-config.default.json (git-tracked), updated server.ts, updated index.ts, updated Dockerfile, updated docker-compose.yml.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP client → PUT /api/config | Untrusted JSON body from browser/curl crosses into server.ts file-write path |
| Host filesystem → container bind mount | ./pipeline is writable by any process with volume access; ACTIVE_PIPELINE_CONFIG_PATH must be deterministic |
| Docker image build → /app/config | Default template baked into image; immutable after build |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-17-01 | Tampering | PUT /api/config write destination | mitigate | D-04: write path is now the hard-coded module-level ACTIVE_PIPELINE_CONFIG_PATH constant; resolveConfigPath() (which could be influenced by PIPELINE_CONFIG_PATH/INPUT_PATH env vars) is removed from the write path. Attacker cannot redirect writes by injecting env vars into a running container without already having container exec. |
| T-17-02 | Tampering | seedDefaultConfig() destination | mitigate | Destination is ACTIVE_PIPELINE_CONFIG_PATH from constants.ts (env-overridable but only at container start by the operator, not by HTTP request). Source is DEFAULT_CONFIG_PATH which is baked into the image. No user-controlled input reaches either path. |
| T-17-03 | Information Disclosure | config/pipeline-config.default.json in git | accept | Template contains only styling preferences (colors, fonts, layout). No secrets, no PII. Appropriate to ship in a public repo. |
| T-17-04 | Elevation of Privilege | Seed-copy clobbering existing user config | mitigate | D-06: seedDefaultConfig() checks existsSync(ACTIVE_PIPELINE_CONFIG_PATH) first and returns immediately if the file exists. The user's saved config is never overwritten by the seed hook. |
| T-17-05 | Tampering | :ro studio volume removed from api-server | mitigate | D-02: removing the mount eliminates a potential write surface where an api-server process with write access could have modified studio source files. The new mount surface is ./pipeline only (already present). |
| T-17-06 | Spoofing | JSON body in PUT /api/config | mitigate | Existing validatePipelineConfig() Zod schema validation (carried from Phase 16) rejects malformed configs before they are written. sanitizeTitles() HTML-escapes title text (CR-02). Both checks remain on the refactored single-write path. |
| T-17-SC | Tampering | npm install during future builds | accept | No new npm packages added in this plan. config/pipeline-config.default.json is a static JSON file with zero dependencies. |

</threat_model>
