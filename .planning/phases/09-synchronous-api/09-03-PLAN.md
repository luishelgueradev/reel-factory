  <action>
Per D-01 and D-06: Create the API server Docker container and add it to the Docker Compose pipeline.

1. Create services/api-server/Dockerfile:
   - FROM node:22-bookworm-slim (lighter than the base-node image since we don't need Chrome/FFmpeg)
   - Install docker-cli (for Docker-in-Docker socket access — the API container needs to run sibling containers)
   - WORKDIR /app
   - COPY package*.json ./
   - RUN npm ci --only=production
   - COPY src/ ./src/
   - COPY tsconfig.json ./
   - RUN npx tsx --version (verify TypeScript works)
   - EXPOSE 3000
   - CMD ["npx", "tsx", "src/index.ts"]
   - NOTE: The container needs access to /var/run/docker.sock to manage sibling containers. This is mounted via docker-compose.yml volumes.

2. Create services/api-server/src/routes/health.ts:
   - GET /health endpoint returning { status: "ok", timestamp: new Date().toISOString(), uptime_seconds: process.uptime() }
   - Simple liveness check — no dependency checks (those could hang)

3. Update services/api-server/src/index.ts to:
   - Import and mount healthRouter at /health
   - Add /health endpoint before /process and /artifacts routes

4. Update docker-compose.yml to add api-server service:
   ```yaml
   # API Server: exposes processing pipeline via REST API
   api-server:
     <<: *pipeline-common
     build:
       context: ./services/api-server
     depends_on:
       base-node:
         condition: service_completed_successfully
     ports:
       - "${API_PORT:-3000}:3000"
     volumes:
       - ./pipeline:/data/pipeline
       - /var/run/docker.sock:/var/run/docker.sock
     environment:
       - PIPELINE_JOB_ID=${PIPELINE_JOB_ID:-}
       - PROCESS_TIMEOUT_MS=${PROCESS_TIMEOUT_MS:-600000}
       - PORT=3000
       - NODE_ENV=production
   ```
   - Key: Docker socket mount at /var/run/docker.sock for sibling container management
   - ports: expose API on configurable port (default 3000)
   - Same pipeline volume mount as all other services for artifact access

5. Verify the Dockerfile builds: `docker compose build api-server`

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → API Container | HTTP request boundary (file upload + JSON response) |
| API Container → Docker Daemon | Docker socket mount allows container management |
| API Container → Pipeline Volume | Shared filesystem for artifact reading |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-08 | Elevation of Privilege | Docker socket mount | mitigate | Docker socket access allows creating/removing containers; API container runs as non-root user; no host filesystem mount beyond /data/pipeline |
| T-09-09 | Tampering | Container image substitution | accept | Attacker with Docker socket access could inject malicious images — accepted risk for v1 (single-user, local deployment) |
| T-09-10 | Information Disclosure | Error messages in responses | mitigate | Pipeline step errors include step name and manifest error_message but NOT container logs or stack traces in production (NODE_ENV=production) |
</threat_model>
